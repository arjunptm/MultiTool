# Codex Working Context for MultiTool

Use this file as quick context when starting a new Codex chat for this repo.
It complements `prompt.txt`: `prompt.txt` describes the app architecture and coding
style, while this file describes collaboration, Git/GitHub workflow, and project
history from prior AI-assisted work.

## User Preferences

- The user usually uses Git Bash, not PowerShell. When giving terminal commands,
  prefer Git Bash syntax unless the user specifically asks for PowerShell.
- The project runs from `W:\MultiTool`.
- Python virtual environment activation in Git Bash:

```bash
source .venv/Scripts/activate
python app.py
```

- Keep explanations practical and beginner-friendly. The user is learning good
  Git/GitHub/Codex workflow while building the app.

## Git Workflow

- Do not work directly on `main` for feature development.
- Use feature branches prefixed with `AI-`, for example:

```bash
git checkout -b AI-sign-pdf-tool
```

- Keep `main` stable and runnable.
- Before committing, check:

```bash
git status
git diff
```

- Preferred commit style: one clear, meaningful commit per coherent feature/fix
  unless the task naturally needs multiple commits.

## GitHub Workflow

- GitHub CLI (`gh`) is installed and authenticated as the user's personal GitHub
  account: `arjunptm`.
- The repo remote is:

```text
git@github.com:arjunptm/MultiTool.git
```

- Use GitHub issues for meaningful feature work, bugs, polish tasks, refactors,
  or future ideas.
- When the user brings a new idea, feature, bug, or polish request, first check
  existing GitHub issues for exact or similar matches before planning or coding:

```bash
gh issue list --state open --limit 100
gh issue list --state all --search "<keywords>"
```

- If a matching issue exists, discuss how the new request fits it and whether
  the existing issue should be used, refined, split, or superseded.
- If no suitable issue exists, create a focused issue before or alongside the
  implementation plan.
- Use pull requests before merging feature branches into `main`, even for solo
  work, because PRs provide a useful review checkpoint.
- Include `Closes #<issue-number>` in PR bodies when the PR should close an
  issue on merge.
- It is acceptable for Codex to use `gh` to create issues, create PRs, inspect
  PR state, and merge PRs when the user explicitly asks.
- When creating or editing GitHub issue/PR bodies that contain markdown
  backticks, tag patterns like `v*`, or other shell-sensitive characters, use a
  safe multiline body approach such as a PowerShell single-quoted here-string
  assigned to `$body`, then pass `--body $body`. Avoid embedding escaped
  backticks directly in a one-line `gh ... --body "..."` command, because that
  can mangle characters in Git Bash/PowerShell contexts.

## First AI-Assisted Feature Already Completed

- Feature branch: `AI-sign-pdf-tool`
- Issue: `#1 Add visual PDF signing tool`
- Pull request: `#2 Add visual PDF signing tool`
- PR #2 was merged into `main`.
- Issue #1 closed automatically through `Closes #1`.

The completed feature added a standalone `Sign PDF` tool under Documents.
It supports:

- opening one PDF
- navigating pages
- uploading a signature image
- drawing a signature with the mouse
- moving the signature visually
- resizing via edge handles
- proportional scaling via the bottom-right handle
- rotating via a top handle
- saving a signed copy without overwriting the original

It is visual signing only, not cryptographic/digital certification.

## Windows Packaging

- Issue #5 covered local/manual Windows executable packaging and is closed.
- Issue #14 covered tagged GitHub Actions release automation and is closed.
- The first automated release tag, `v0.1.0`, published
  `MultiTool-v0.1.0-windows.zip`.
- Issue #17 tracks future evaluation of macOS/Linux support.
- The first local packaging pass uses PyInstaller with a folder-based build.
- `MultiTool.spec` is the committed PyInstaller build recipe.
- `requirements-build.txt` installs normal app dependencies plus PyInstaller.
- `build/` and `dist/` are generated outputs and should not be committed.

Useful local packaging commands:

```bash
source .venv/Scripts/activate
pip install -r requirements-build.txt
pyinstaller --noconfirm --clean MultiTool.spec
./dist/MultiTool/MultiTool.exe
```

Useful tagged release commands:

```bash
git checkout main
git pull
git tag v0.1.0
git push origin v0.1.0
```

## End-of-Session Wrap-Up

When the user says something like "let's run a wrap-up check", treat it as a
request to prepare the repo and project tracking for a stopping point.

Wrap-up checklist:

- Check `git status` and summarize uncommitted, staged, and untracked changes.
- Review whether README, `prompt.txt`, and `codex.txt` need updates for the work
  completed in the session.
- Run appropriate lightweight verification commands for the touched area when
  practical.
- Check related GitHub issues and pull requests for status, missing links, or
  follow-up notes.
- Prompt the user before committing, pushing, creating a PR, merging, closing
  issues, or making any other state-changing GitHub/Git action.
- Leave the user with a clear summary of what is done, what remains, and the
  recommended next action.
